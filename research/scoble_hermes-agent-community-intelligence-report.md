## **The Learning Loop — The Core Innovation**

This is what separates Hermes from every other open-source agent. It's a **closed learning loop** that makes the agent genuinely smarter over time without any manual intervention.

### **1\. Skill Creation**

After completing a complex task, Hermes autonomously creates a "skill" — a reusable procedure it can call on in future sessions. Skills are stored in \~/.hermes/skills/ and are compatible with the **agentskills.io** open standard, meaning they're portable and shareable.

### **2\. Skill Self-Improvement**

Skills don't just get created — they improve themselves during use. When Hermes executes a skill and encounters a better approach, it updates the skill definition.

### **3\. Memory Nudges**

The agent periodically nudges itself to persist important knowledge to MEMORY.md and USER.md. This happens autonomously — you don't have to tell it to remember things.

### **4\. Cross-Session Recall**

All CLI and messaging sessions are stored in SQLite (\~/.hermes/state.db) with FTS5 full-text search. The agent can search its own past conversations using the session\_search tool, with Gemini Flash summarization of results.

bash

hermes sessions list    \# Browse past sessions

### **5\. User Modeling (Honcho)**

Honcho dialectic user modeling builds a deepening model of who you are across sessions. It tracks your preferences, working style, and corrections. Enable it with:

bash

hermes honcho setup

## **Memory System — Deep Dive**

Hermes has a two-layer memory architecture:

### **Layer 1: Persistent Memory (Always in Context)**

Two files, always injected into the system prompt:

**MEMORY.md** — Agent's personal notes

• Key facts, project details, learned preferences

• \~800 tokens capacity (\~2,200 characters)

• Managed by the agent autonomously

**USER.md** — User profile

• Who you are, your preferences, working style

• \~500 tokens capacity (\~1,375 characters)

• Updated by the agent as it learns about you

**Best practice for memory entries:**

\# Good: Packs multiple related facts  
User runs macOS 14 Sonoma, uses Homebrew, has Docker Desktop. Shell: zsh with oh-my-zsh. Editor: VS Code with Vim keybindings.  
   
\# Good: Specific, actionable conventions  
Project \~/code/api uses Go 1.22, sqlc for DB queries, chi router. Run tests with 'make test'. CI via GitHub Actions.  
   
\# Bad: Too vague  
User has a project.

When memory exceeds 80% capacity, the agent consolidates entries automatically.

### **Layer 2: Session Search (Unlimited History)**

All sessions stored in SQLite with FTS5 full-text search. The agent queries this when it needs to recall something from weeks ago that isn't in active memory.

| Feature | Persistent Memory | Session Search | |---------|------------------|----------------| 

| Capacity | \~1,300 tokens total | Unlimited |   
| Speed | Instant (in system prompt) | Requires search \+ LLM summarization |   
| Use case | Key facts always available | Finding specific past conversations |   
| Management | Autonomously curated | Automatic — all sessions stored |

### **Security Scanning**

Memory entries are scanned before being accepted:

• Prompt injection patterns blocked

• Credential exfiltration patterns blocked

• SSH backdoor patterns blocked

• Invisible Unicode characters blocked

## **Cron Scheduling — Automated Agents**

Hermes has a built-in cron scheduler that lets you set up automated tasks in plain English, with delivery to any connected platform.

### **How It Works**

bash

### **Per-Job Configuration**

### **Technical Details**

• Jobs stored in SQLite for persistence across restarts

• Atomic writes prevent data loss on crash

• Thread context preserved for deliver: origin jobs

• Duplicate send prevention for auto-delivery targets

## **Subagent Delegation & Parallelization**

Hermes can spawn isolated subagents to work on parallel workstreams. This is one of its most powerful features for complex tasks.

### **How Delegation Works**

Parent Agent  
├── Subagent 1: "Research competitor pricing"  
├── Subagent 2: "Write the technical spec"  
└── Subagent 3: "Set up the test environment"

Key design principle: **Context isolation IS the key feature.** The parent never sees the child's full conversation — only the summary. This prevents "context poisoning" where delegated work floods the parent's context window.

• **Auto-reload** — MCP tools reload when config changes, no restart needed

• **Tool filtering** — prevent context bloat from too many MCP tools

### **MCP vs Skills Trade-off**

MCP tools consume context space. For frequently-used capabilities, creating a skill (CLI-based) is more context-efficient. Hermes supports both approaches.

## **Security Architecture**

Hermes has a multi-layer security architecture:

### **1\. Tirith Pre-Exec Command Scanning**

Static analysis of terminal commands before execution. Analyzes commands for dangerous patterns before running them.

### **2\. Smart Approvals**

Codex-inspired approval system that learns which commands are safe and remembers your preferences. You approve once, it remembers.

bash

\# During setup, configure approval settings  
hermes setup    \# Includes approval configuration

### **3\. PII Redaction**

When enabled, personally identifiable information is automatically scrubbed before sending context to LLM providers:

yaml

privacy:  
  redact\_pii: true

### **4\. Container Isolation**

Docker backend with explicit opt-in for workspace mounting — host directories are never auto-mounted.

### **5\. Environment Variable Isolation**

Hermes strips its own provider/gateway/tool env vars from all subprocess environments, preventing credential leakage to child processes.

### **6\. Memory Security Scanning**

Memory entries are scanned for:

• Prompt injection patterns

• Credential exfiltration patterns

• SSH backdoor patterns

• Invisible Unicode characters

### **7\. Supply Chain Security**

From @Teknium:

*"Package and dependency versions are tightly constrained ranges now and our PRs get tested for a variety of supply chain attack vectors when submitted during CI now."*

## **Research & RL Training (Atropos)**

Hermes is built by model trainers, for model trainers. It includes a full research pipeline:

### **Batch Trajectory Generation**

Generate training data at scale from agent interactions.

### **Atropos RL Integration**

Hermes integrates with **tinker-atropos**, NousResearch's RL training framework:

bash

\# Optional RL training setup  
git submodule update \--init tinker-atropos  
uv pip install \-e "./tinker-atropos"

### **Agentic On-Policy Distillation (OPD)**

New in v0.3.0: A training environment for distilling agent policies. The agent generates trajectories that are used to train better versions of itself.

### **Trajectory Compression**

Compress multi-step trajectories for training tool-calling models. This is how NousResearch generates training data for the next generation of Hermes models.

**The meta-loop:** Hermes generates training data → NousResearch trains new Hermes models → New models make Hermes better → Hermes generates better training data.
